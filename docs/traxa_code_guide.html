0. مفاهيم Flutter اللي محتاج تفهمها قبل القراءة

0.1 ما هو الـ Widget؟

كل حاجة تشوفها على الشاشة في Flutter هي Widget: زرار، نص، صورة، حتى المساحة الفارغة (Padding/SizedBox). الـ Widgets بتتركب في شجرة (Widget Tree) — الأب يحتوي على أولاد، الأولاد لهم أولاد، وهكذا. 

Scaffold              ← Widget الـ App الأساسي (يعمل layout للـ AppBar + Body + BottomNav)
 └─ Column            ← يرتب أولاده عمودياً
     ├─ Text(...)     ← نص
     ├─ SizedBox(h:8) ← مسافة فاضية ارتفاعها 8
     └─ ElevatedButton ← زرار

0.2 StatelessWidget vs StatefulWidget

StatelessStateful
ما عندوش بيانات تتغير. ترسم نفس الـ UI كل مرة. عنده State ممكن يتغير → الـ Widget يعيد البناء.
أمثلة: شعار، عنوان ثابت، Container زخرفة. أمثلة: Form، Counter، شاشة بتحمل بيانات من API.

الـ StatefulWidget لما setState(() {}) ينادى، Flutter يعيد بناء الـ widget بالبيانات الجديدة.

0.3 الـ BuildContext

الـ context هو المرجع لمكان الـ Widget في الشجرة. ممكن منه نوصل للـ Theme، للـ MediaQuery (المقاسات)، أو لأي Cubit/Provider فوقه عبر context.read<X>() أو context.watch<X>().

0.4 Container — أهم Widget هنشوفه كتير

Container هو "صندوق" بيتلف widget. بيقدر يضيف:

  • padding — مسافة داخلية حول المحتوى.
  • margin — مسافة خارجية حواليه.
  • decoration — لون، gradient، border، borderRadius، shadow.
  • width / height — مقاسات.
  • alignment — مكان المحتوى جواه.

0.5 الـ State Management — BLoC / Cubit

المشكلة: لو الـ data في شاشة 1 محتاجين نعرضها في شاشة 5، إزاي ننقلها؟ الحل التقليدي (تمرير constructor parameters) صعب لما الشجرة كبيرة.

الحل: Cubit (نسخة مبسطة من BLoC). الـ Cubit class عنده state، الـ widgets تستمع له (BlocBuilder) وتعيد البناء عند تغيره. أي widget يقدر ينادي method على الـ Cubit عبر context.read<MyCubit>().doSomething(). 

class CounterCubit extends Cubit<int> {
  CounterCubit() : super(0);    // ← الحالة الابتدائية
  void increment() => emit(state + 1);  // emit بيرسل state جديدة
}

// في UI:
BlocBuilder<CounterCubit, int>(
  builder: (context, count) => Text('$count'),

0.6 async / await و Future

أي حاجة بتاخد وقت (HTTP request، قراءة ملف) هي Future. كلمة await تستنى لحد ما تخلص. الدالة اللي فيها await لازم تبقى async. 

Future<String> getName() async {

  return 'Ali';
}
// استخدام: final name = await getName();

0.7 ValueNotifier

متغير بسيط بقيمة واحدة، أي حد بيستمع له يتنبه لما تتغير. أخف من Cubit للحالات الـ ultra-local (مثلاً: هل الـ password مخفي ولا لأ؟).

1. main.dart — البداية

1.1 main() — أول دالة تشتغل

Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await ServerConfig.load();
  HttpOverrides.global = MyHttpOverrides();
  await initServiceLocator();
  runApp(const MyApp());
}
سطر سطر:
  1. WidgetsFlutterBinding.ensureInitialized() — Flutter بيستخدم محرك C++ تحت السطح. الـ binding هو الجسر بين Dart والمحرك. لازم نفعّله أول حاجة لو هنعمل async work قبل runApp() (زي قراءة من SharedPreferences).
  2. await ServerConfig.load() — يقرأ URL السيرفر اللي المستخدم حفظه آخر مرة من SharedPreferences. (مثلاً https://192.168.1.17 لو شغال محلي).
  3. HttpOverrides.global = MyHttpOverrides() — يجبر التطبيق إنه يثق بـ shadhada SSL للسيرفر اللي محدد (مفيد للـ self-signed certs).
  4. await initServiceLocator() — يسجل كل الـ services والـ cubits في GetIt (الـ Container الـ DI). شوف القسم اللي جاي.
  5. runApp(MyApp()) — يبدأ تشغيل الـ Flutter app ويرسم MyApp على الشاشة.

1.2 MyHttpOverrides

class MyHttpOverrides extends HttpOverrides {
  @override
  HttpClient createHttpClient(SecurityContext? context) {
    return super.createHttpClient(context)
      ..badCertificateCallback = (cert, host, port) {
        return host == Uri.parse(AppConstants.baseUrl).host;
      };
  }
}
ليه موجود؟ لو السيرفر شغال على HTTPS بشهادة self-signed (مثلاً سيرفر تطوير محلي)، Flutter بيرفضها افتراضياً. الكود ده بيقول "تجاهل الخطأ لو الـ host هو السيرفر بتاعنا فقط". في الـ production السيرفر يستخدم شهادة حقيقية فالكود ده مش بيؤثر.
الـ .. (cascade operator): طريقة لكتابة multiple statements على نفس الـ object بدون تكرار اسمه. obj..a()..b() = obj.a(); obj.b();

1.4 _AuthGate — البواب اللي بيقرر فين تروح

class _AuthGate extends StatelessWidget {
  Widget build(BuildContext context) {
    return BlocConsumer<AuthCubit, AuthState>(
      listenWhen: (prev, curr) => curr.isAuthenticated && curr.user != null &&
          (prev.user?.id != curr.user?.id || !prev.isAuthenticated),
      listener: (context, authState) {
        context.read<DataCubit>().fullReload();
      },
      builder: (context, authState) {
        if (authState.isAuthenticated && authState.user != null) {
          if (authState.user!.isDoctor || authState.user!.isTeachingAssistant) {
            return const DoctorScreen();
          }
          return const StudentScreen();
        }
        return const LoginScreen();
      },
    );
  }
}
  • BlocConsumer = BlocBuilder + BlocListener معاً.
    • listener: يشغل side-effect (مش بيغير UI). هنا: لما المستخدم يسجل دخول ناجح → DataCubit يعمل reload للبيانات.
    • builder: يعيد بناء الـ UI حسب الحالة.
  • listenWhen condition: نشغل الـ listener فقط لما يكون فعلاً متعمل login جديد (مش كل rebuild).
  • builder logic: لو فيه user → روح لشاشته. لو لأ → LoginScreen.

دوال AuthCubit

init() / checkAuth()

Future<void> checkAuth() async {
  final result = await _repo.restoreSession();
  result.when(
    success: (session) async {
      if (session == null) {
        emit(AuthState.initial());
        return;
      }
      emit(AuthState.success(session.user, session.token));
      await WebSocketService.instance.connect();

الخطوات:

  2. result.when(success, failure) — نتعامل مع الحالتين.

  4. لو موجودة: emit success → الـ AuthGate يفتح الشاشة المناسبة + نشغل WebSocket.

2.3 ThemeCubit — الـ Dark/Light Mode

class ThemeCubit extends Cubit<ThemeState> {
  ThemeCubit() : super(ThemeState.initial()) {
    _loadTheme();   // ← يجيب الـ theme المحفوظ من SharedPrefs
  }

  Future<void> setTheme(ThemeMode mode) async {
    final prefs = await SharedPreferences.getInstance();
    await prefs.setString('theme_mode', mode == ThemeMode.dark ? 'dark' : 'light');
    emit(ThemeState(themeMode: mode));
  }

  void toggleTheme() {
    final newMode = state.themeMode == ThemeMode.dark
        ? ThemeMode.light
        : ThemeMode.dark;
    setTheme(newMode);
  }
}
  • Constructor: يحمّل الـ theme المحفوظ تلقائياً عند الإنشاء.
  • setTheme: يحفظ في SharedPrefs ثم emit حالة جديدة → MaterialApp يلوّن الـ UI بالـ theme الجديد.
  • toggleTheme: يقلب بين dark و light. زرار الـ ThemeToggleButton ينادي عليها.

3. WebSocketService — الـ Realtime

3.1 Singleton Pattern

class WebSocketService {
  static WebSocketService? _instance;
  static WebSocketService get instance => _instance ??= WebSocketService._();
  WebSocketService._() { _initStreams(); }
}
ليه Singleton؟ مفيش معنى لفتح أكثر من WS connection. لما أي كود ينادي WebSocketService.instance يحصل على نفس النسخة في كل التطبيق.

??= = "لو null اعمل assignment، غير كده سيبه". فالـ instance يتعمل مرة واحدة فقط.

3.2 الـ Streams — قنوات البث

StreamController<Map<String, dynamic>>? _gradeUpdateController;

Stream<Map<String, dynamic>> get gradeUpdateStream {
  if (_gradeUpdateController == null || _gradeUpdateController!.isClosed) {
    _initStreams();
  }
  return _gradeUpdateController!.stream;
}
StreamController = مصدر للـ stream. الكود اللي بيستقبل messages من السيرفر يستخدم _gradeUpdateController.add(data) ليبث.
Stream = القناة اللي الـ UI يستمع لها. مثل اشتراك في قناة يوتيوب: كل ما تنزل حلقة (event) المشتركين يشوفوها.
broadcast() = يسمح بأكثر من listener (المعتاد stream له listener واحد فقط).

3.3 connect()

Future<void> connect() async {
  if (_isConnected && _channel != null) return;       // (1)

  final token = await SecureStorageService.getToken();   // (2)
  final userData = await SecureStorageService.getUserData();
  if (token == null || token.isEmpty || userData == null) return;

  final user = jsonDecode(userData);
  final userRole = user['role'] ?? 'guest';
  String clientType = userRole == 'doctor' ? 'mobile-doctor' : 'mobile-student';

  final wsUrl = '${AppConstants.wsUrl}?type=mobile';     // (3)

  final socket = await WebSocket.connect(wsUrl, customClient: httpClient)
      .timeout(const Duration(seconds: 12));             // (4)

  _channel = IOWebSocketChannel(socket);
  _isConnected = true;

  _channel!.stream.listen(                                // (5)
    (message) => _handleMessage(message),
    onDone: () { _isConnected = false; _scheduleReconnect(); },
    onError: (error) { _isConnected = false; _scheduleReconnect(); },
  );

  sendMessage({                                           // (6)
    'type': 'REGISTER',
    'clientType': clientType,
    'userId': user['id'],
  });

  _startHeartbeat();                                      // (7)
}
  1. لو متصل بالفعل → ارجع. (تجنب اتصال مزدوج).
  2. اقرأ الـ token والـ user من SecureStorage. لازم يكون فيه login عشان نتصل.
  3. جهز الـ URL: wss://traxa-system.online?type=mobile.
  4. اتصل مع timeout 12 ثانية.
  5. استمع للرسائل الجاية من السيرفر. لو الاتصال انقطع → جدول reconnect.
  6. أرسل أول رسالة REGISTER لتعريف نفسك للسيرفر.
  7. ابدأ heartbeat (نبضات كل 25 ثانية للحفاظ على الاتصال).

      break;
    case 'PING':
      sendMessage({'type': 'PONG'});
      break;
    ...
  }
}
كل نوع رسالة عنده stream خاص. الـ widgets المستمعة بتاخد فقط النوع اللي يهمها بدل ما تستمع لكل شيء. مثلاً شاشة الدرجات تستمع لـ gradeUpdateStream فقط.

3.5 _scheduleReconnect — Exponential Backoff

void _scheduleReconnect() {
  if (_isConnected) return;
  final delay = Duration(seconds: min(5 * (_reconnectAttempts + 1), 30));
  _reconnectTimer = Timer(delay, () {
    _reconnectAttempts++;
    connect();
  });
}
الفكرة: لو الاتصال انقطع، استنى 5 ثواني وحاول. لو فشل تاني، 10 ثواني. ثم 15. حتى 30 ثانية كحد أقصى. ده اسمه "Exponential Backoff" — يقلل ضغط السيرفر لو فيه مشكلة كبيرة.

4. Repositories — وسيط بين الـ Cubit والـ API

4.1 BaseRepository.guard() — أهم دالة

Future<Result<T>> guard<T>(Future<T> Function() action, {String? tag}) async {
  try {
    return Success(await action());
  } on AppException catch (e) {
    return Failure(e);
  } on SocketException catch (e, s) {
    return Failure(NetworkException('No internet connection', cause: e, stack: s));
  } on http.ClientException catch (e, s) {
    return Failure(NetworkException(e.message, cause: e, stack: s));
  } on async.TimeoutException {
    return Failure(const TimeoutException());
  } catch (e, s) {
    return Failure(UnknownAppException(e.toString(), cause: e, stack: s));
  }
}
اللي بتعمله:
  1. تشغل الـ action (مثلاً HTTP call).
  2. لو نجح: ترجع Success(value).
  3. لو نزل exception، تمسكه وتحوله لـ Failure(typed exception).
ليه مهمة؟
- الـ Cubit ما يحتاجش يكتب try/catch في كل دالة.
- الأخطاء بقت typed (NetworkException، TimeoutException...) فالـ UI يعرف يفرّق.
- الـ Result<T> يفرض على الكود يعالج الحالتين (sealed class بتعني المترجم هيشتكي لو نسيت Failure).

4.2 AuthRepository.studentLogin()

Future<Result<({User user, String token})>> studentLogin({...}) async {
  return guard(() async {
    final response = await ApiService.studentLogin(studentId, password, isFirstLogin);
    final outcome = fromLegacyResponse(response);
    return outcome.when(
      success: (data) async {
        final json = Map<String, dynamic>.from(data['student']);
        json['role'] = 'student';
        json['userType'] = 'student';
        json['username'] ??= json['student_id']?.toString();
        final user = User.fromJson(json);
        await _persistSession(user, data['token'] as String, fallbackType: 'student');
        return (user: user, token: data['token'] as String);
      },
      failure: (e) => throw e,
    );
  }, tag: _tag);
}
  1. تشغل HTTP login عبر ApiService.
  2. fromLegacyResponse — تحول الـ {'success': true, ...} القديم إلى Result.
  3. تضيف role/userType (مش بيرجعوا من السيرفر للطلاب).
  4. تنشئ User object.
  5. تحفظ في SecureStorage (الـ session).
  6. ترجع tuple = (user: ..., token: ...).
Record (Tuple) في Dart 3: (user: u, token: t) = object صغير بـ named fields بدون الحاجة لتعريف class. الـ Cubit يستخدمه: session.user, session.token.

4.3 AttendanceRepository.waitForCameraStarted()

Future<bool> waitForCameraStarted({
  required Object sessionId,
  required String token,
  Duration timeout = const Duration(seconds: 90),
  Duration interval = const Duration(milliseconds: 500),
}) async {
  final deadline = DateTime.now().add(timeout);
  while (DateTime.now().isBefore(deadline)) {
    try {
      final res = await http.get(...).timeout(const Duration(seconds: 3));
      if (res.statusCode == 200) return true;
    } catch (_) {}
    await Future.delayed(interval);
  }
  return false;
}
Polling Pattern:
  1. كل نص ثانية، اتصل بالسيرفر واسأل: "صفحة الكاميرا اتفتحت على الـ host؟"
  2. لو رد 200 → نعم، ارجع true.
  3. لو فضلت 90 ثانية بدون رد → ارجع false.
ليه 90 ثانية؟ لما السيرفر يطلب فتح الكاميرا، صفحة الـ host تحمّل face-api models من GitHub CDN، اللي ممكن تاخد دقيقة على شبكة الجامعة البطيئة.

5. Services

5.1 SecureStorageService — التخزين المشفر

class SecureStorageService {
  static const _storage = FlutterSecureStorage(
    aOptions: AndroidOptions(encryptedSharedPreferences: true),
    iOptions: IOSOptions(accessibility: KeychainAccessibility.first_unlock),
  );

  static Future<void> saveToken(String token) =>
    _write(_tokenKey, token, 'Token');

  static Future<String?> getToken() =>
    _read(_tokenKey, 'Token');
}
ليه مش SharedPreferences عادي؟
SharedPreferences بتحفظ نص خام يقدر أي حد عنده وصول للجهاز يقراه. SecureStorage بتستخدم:
  • Android: EncryptedSharedPreferences (مشفر AES).
  • iOS: Keychain (hardware-backed).
ده مهم للـ JWT token لأنه يمثل هوية المستخدم.

  • القراءة العادية: لو الـ cache أقدم من ساعة، ارجع null (DataCubit يجلب من السيرفر).
  • القراءة بـ ignoreExpiry=true: ارجع البيانات حتى لو قديمة. يستخدم لما الـ network يفشل (أفضل من شاشة فاضية).

5.3 TokenHolder — Token قابل للمراقبة

class TokenHolder {
  final ValueNotifier<String?> _token = ValueNotifier<String?>(null);

  ValueListenable<String?> get listenable => _token;

  Future<String?> load() async {
    final saved = await SecureStorageService.getToken();
    _token.value = saved;
    return saved;
  }

  Map<String, String> get authHeader =>
    hasToken ? {'Authorization': 'Bearer ${_token.value!}'} : const {};
}
ValueNotifier أبسط من Cubit للحالات الـ single-value. authHeader getter يرجع الـ HTTP header جاهز للاستخدام: http.get(url, headers: tokenHolder.authHeader).

8. حسابات الـ GPA — helpers.dart

8.1 calculateGPA()

double calculateGPA(List<Grade> grades, List<Subject> subjects) {
  double totalPoints = 0;
  int totalCredits = 0;
  for (final grade in grades) {
    if (!grade.isVisible) continue;
    final subject = findSubjectSafely(grade.subjectId, subjects);
    if (subject == null) continue;
    final credits = subject.totalCreditHours;
    totalPoints += grade.gradePoints * credits;
    totalCredits += credits;
  }
  return totalCredits > 0 ? (totalPoints / totalCredits) : 0.0;
}
المعادلة:

GPA = Σ (gradePoints × creditHours) / Σ creditHours

  1. تخطّ الدرجات غير المرئية (admin قفلها).
  2. لكل grade، جيب الـ subject عشان نعرف الكريدتس.
  3. اضرب الـ gradePoints (4.0 لـ A+، 0.0 لـ F) في الكريدتس.
  4. اقسم المجموع على مجموع الكريدتس.

gradePoints (في model Grade)

double get gradePoints {
  if (total >= 96) return 4.0;
  if (total >= 92) return 3.7;
  if (total >= 88) return 3.4;
  ...
  if (total >= 50) return 1.0;
  return 0.0;
}
الـ Scale: 12-point scale (A+/A/A-/B+/B/B-/C+/C/C-/D+/D/D-/F). كل letter له قيمة عددية بين 0.0 و 4.0.

8.2 calculateEarnedCredits()

int calculateEarnedCredits(List<Grade> grades, List<Subject> subjects) {
  int total = 0;
  for (final grade in grades) {
    if (grade.total >= 50 && grade.isVisible) {
      final subject = findSubjectSafely(grade.subjectId, subjects);
      if (subject != null) total += subject.totalCreditHours;
    }
  }
  return total;
}

11. أنماط مهمة (Patterns) في المشروع

11.1 Repository Pattern

الـ Cubits لا تتحدث مباشرة مع HTTP أو storage. كل حاجة تمر عبر Repository. الفايدة:

  • سهل اختبار الـ Cubits بـ FakeRepository.
  • لو غيرنا source البيانات (مثلاً GraphQL بدل REST)، نغير repository فقط.

11.2 Service Locator (GetIt)

getIt.registerLazySingleton<AuthRepository>(() => AuthRepository());

// الاستخدام:
final repo = getIt<AuthRepository>();

بدل ما تمرر AuthRepository من main لكل widget جوه، أي كود يقدر يجيبها من GetIt. lazySingleton = يتعمل أول مرة بس يتم طلبه (مش وقت الـ register).

11.3 Result<T> Pattern

sealed class Result<T> {}
class Success<T> extends Result<T> { final T value; }
class Failure<T> extends Result<T> { final AppException exception; }

بدل ما الدوال ترمي exceptions، ترجع Result. الكود اللي بيستدعيها مضطر يعالج الحالتين (success و failure) لأن الـ sealed بتجبره.

11.4 Singleton with lazy initialization

static WebSocketService? _instance;
static WebSocketService get instance => _instance ??= WebSocketService._();

11.5 Listenable / ValueNotifier للـ Local State

بدل Cubit للحالات الصغيرة جداً (boolean toggle مثلاً)، نستخدم ValueNotifier + ValueListenableBuilder. أخف وأسرع.
